# WiSP Server

HTTP backend that serves processed images to ESP32-based e-paper photo frames.

## Features

- Serves vendor-specific binary images to ESP32 firmware over HTTP
- Full image processing pipeline: EXIF rotation, aspect-ratio crop, brightness / contrast / gamma / hue / saturation / blur, dithering-based color reduction, timestamp overlay, and explicit rotation
- Floyd-Steinberg, Bayer, and Sierra3 dithering algorithms
- Multiple catalog providers: local filesystem, HTTP URL, and color-bar test pattern
- NDJSON streaming for large photo catalogs
- Concurrent image scanning using all available CPU cores
- SQLite (development) and MySQL (production) database backends, switched via config
- Per-image visibility toggle via REST API
- Randomized image delivery with anti-repeat logic

## Hardware Requirements

- A Waveshare e-paper display (see [Supported Display Models](#supported-display-models))
- ESP32 microcontroller running WiSP firmware
- Server host: Linux, macOS, or Docker (any platform that runs Go)

## Getting Started

> **Monorepo users:** to run the full stack (API + frontend together), start from the [repo root](../README.md) instead. The commands below work from within `api/` for API-only development.

### 1. First-time setup

```bash
make dev-setup
```

Creates `config/config.yaml` (SQLite, port 9002) and `config/service.yaml` and places sample photos in the configured source directory. Both files are gitignored.

### 2. Start the server

```bash
docker compose up
```

The API server starts on port 9002 with air hot-reload enabled.

### 3. Scan images into the database

```bash
docker compose exec api ./wisp catalog scan
# With verbose output:
docker compose exec api ./wisp catalog scan --verbose
```

### 4. Verify the server is running

```bash
curl http://localhost:9002/health
```

### Building locally (without Docker)

Prerequisites: Go 1.26+

```bash
make build
./wisp web run
```

### Testing

```bash
go test ./...
# With race detector:
go test -race ./...
# With HTML coverage report:
make test
```

### Linting

```bash
make lint-local
```

## Configuration

Both configuration files are gitignored and auto-generated by `make dev-setup`. Example files live in `config/`.

### `config/config.yaml`

```yaml
log_level: INFO        # DEBUG | INFO | WARN | ERROR
port: 9002
database:
  driver: sqlite       # sqlite | mysql
  dsn: /data/wisp.db
  log: false           # true logs every GORM query
```

For MySQL, set `driver: mysql` and a DSN such as:

```
user:password@tcp(host:3306)/dbname?parseTime=true&charset=utf8mb4
```

### `config/service.yaml`

```yaml
catalog:
  - key: photos
    type: file         # file | http | colorbar
    file:
      src_path: /mnt/wisp

displays:
  - name: my-frame
    mac_address: dev          # URL key → /pf/dev/image/random.bin
    model: ws7in3e            # see Supported Display Models
    orientation: landscape    # landscape | portrait
    flip: false
    show_timestamp: true
    color_reduction:
      type: floysteinberg     # floysteinberg | bayer | sierra3
      size: 4
      strength: 0.8
    catalog:
      - key: photos
    filters:                  # optional per-image adjustments
      - type: gamma
        properties:
          value: "1.2"
```

`mac_address` becomes the `:displayKey` path parameter. Use `dev` for local testing; set it to the ESP32 MAC address in production (the firmware sends requests to `/pf/{MAC}/image/random.bin`).

## API Reference

Base URL: `http://localhost:9002`

| Method | Path | Description |
|--------|------|-------------|
| GET | `/health` | Liveness probe |
| GET | `/version` | Build version and commit hash |
| GET | `/catalogs` | List all configured catalogs |
| GET | `/catalog/:key/images` | NDJSON stream of images in catalog |
| GET | `/catalog/:key/image/:id` | Single image thumbnail by ID |
| POST | `/catalog/selected/_toggle-visibility` | Toggle show/hide for a list of image IDs |
| GET | `/pf/:displayKey/image/random.bin` | Random image for ESP32 (vendor binary) |
| GET | `/pf/:displayKey/image/random.jpg` | Random image for ESP32 (JPEG) |

## Usage

### CLI commands

```bash
./wisp web run                    # Start API server (default port 9002)
./wisp catalog scan               # Index images from all configured providers
./wisp catalog scan --verbose     # Index images with per-file logging
./wisp catalog purge-orphans      # Remove DB records with no matching source file
./wisp catalog list               # List all catalogs
./wisp catalog list-images        # List images in default catalog
./wisp system prune               # Drop and recreate the Image table
./wisp version                    # Show build version and commit hash
```

### Catalog providers

| `type` | Description |
|--------|-------------|
| `file` | Scans a local directory for JPEG, PNG, GIF, BMP, HEIF, TIFF, and WebP files |
| `http` | Fetches images from an HTTP URL |
| `colorbar` | Generates a color-bar test pattern (useful for display tuning) |

### Supported display models

| Model key | Size | Colors | Resolution |
|-----------|------|--------|------------|
| `ws4in0e` | 4.0″ | 7-color | 640 × 480 |
| `ws5in65f` | 5.65″ | 7-color | 600 × 448 |
| `ws7in3f` | 7.3″ | 7-color | 800 × 480 |
| `ws7in3e` | 7.3″ | 7-color | 800 × 480 |
| `ws13in3e` | 13.3″ | 7-color | 1600 × 1200 |
| `ws13in3k` | 13.3″ | 3-color | 1600 × 1200 |

### Image processing pipeline

Each image served to an ESP32 passes through these steps in order:

1. EXIF rotation correction
2. Aspect-ratio crop to display dimensions
3. Configurable filters: brightness, contrast, gamma, hue, saturation, blur
4. Color reduction with dithering (Floyd-Steinberg, Bayer, or Sierra3)
5. Timestamp overlay (optional — controlled by `show_timestamp`)
6. Explicit rotation (optional)
